The Marble API uses conventional HTTP status codes to indicate the success or failure of requests. All error responses follow a consistent JSON format to help you debug issues.

## Error response format

Error responses include an `error` field with a short description, and often a `message` field with additional details:

```json
{
  "error": "Post not found",
  "message": "The requested resource does not exist"
}
```

Some errors also include a `details` array for validation errors:

```json
{
  "error": "Invalid query parameters",
  "message": "Validation failed",
  "details": [
    {
      "field": "limit",
      "message": "Expected number, received string"
    }
  ]
}
```

## HTTP status codes

The Marble API uses the following HTTP status codes:

### 2xx Success

| Code | Description |
|------|-------------|
| `200 OK` | Request succeeded, resource returned |
| `201 Created` | Resource created successfully |

### 3xx Redirection

| Code | Description |
|------|-------------|
| `308 Permanent Redirect` | Non-versioned route redirected to `/v1/*` |

### 4xx Client errors

| Code | Description |
|------|-------------|
| `400 Bad Request` | Invalid request parameters or body |
| `401 Unauthorized` | Missing, invalid, disabled, or expired API key |
| `403 Forbidden` | Valid API key but insufficient permissions |
| `404 Not Found` | Resource does not exist |
| `409 Conflict` | Resource already exists (e.g., duplicate slug) |
| `429 Too Many Requests` | Rate limit or usage limit exceeded |

### 5xx Server errors

| Code | Description |
|------|-------------|
| `500 Internal Server Error` | Unexpected server error |

## Common errors

### 400 Bad Request

Returned when request parameters or body are invalid.

<CodeGroup>

```json Invalid query parameter
{
  "error": "Invalid query parameters",
  "details": [
    {
      "field": "limit",
      "message": "Expected number, received string"
    }
  ]
}
```

```json Invalid request body
{
  "error": "Invalid request body",
  "message": "Category is required"
}
```

```json Invalid page number
{
  "error": "Invalid page number",
  "details": {
    "message": "Page 10 does not exist.",
    "totalPages": 5,
    "requestedPage": 10
  }
}
```

</CodeGroup>

### 401 Unauthorized

Returned when authentication fails.

<CodeGroup>

```json Missing API key
{
  "error": "Unauthorized",
  "message": "API key required. Provide via Authorization header or ?key= query parameter"
}
```

```json Invalid API key
{
  "error": "Unauthorized",
  "message": "Invalid API key"
}
```

```json Disabled API key
{
  "error": "Unauthorized",
  "message": "API key is disabled"
}
```

```json Expired API key
{
  "error": "Unauthorized",
  "message": "API key has expired"
}
```

</CodeGroup>

### 403 Forbidden

Returned when you have a valid API key but lack permission for the operation.

```json
{
  "error": "Forbidden",
  "message": "Write operations require a private API key (msk_...). Public keys are read-only."
}
```

This occurs when attempting write operations (`POST`, `PUT`, `PATCH`, `DELETE`) with a public API key (`mpk_...`).

### 404 Not Found

Returned when a requested resource doesn't exist.

<CodeGroup>

```json Post not found
{
  "error": "Post not found",
  "message": "The requested resource does not exist"
}
```

```json Invalid workspace (legacy routes)
{
  "error": "Invalid workspace",
  "message": "The provided workspace key is invalid or does not exist"
}
```

</CodeGroup>

### 409 Conflict

Returned when creating a resource that already exists.

```json
{
  "error": "Slug already in use",
  "message": "A resource with this slug already exists in this workspace"
}
```

### 429 Too Many Requests

Returned when rate limits or usage limits are exceeded.

<CodeGroup>

```json Rate limit exceeded
{
  "error": "Too many requests"
}
```

```json Usage limit exceeded
{
  "error": "Usage limit exceeded",
  "message": "You have reached your API request limit for this billing period. Please upgrade your plan or wait until your usage resets."
}
```

</CodeGroup>

See the [rate limiting](/api/rate-limiting) page for details on handling rate limits.

### 500 Internal Server Error

Returned when an unexpected error occurs on the server.

```json
{
  "error": "Internal server error"
}
```

Or with additional context:

```json
{
  "error": "Internal server error",
  "message": "Failed to fetch resource"
}
```

<Note>
  If you consistently receive 500 errors, please contact support with the request details and timestamp.
</Note>

## Error handling best practices

Implement proper error handling in your application:

<CodeGroup>

```javascript JavaScript
async function fetchPost(slug) {
  try {
    const response = await fetch(
      `https://api.marblecms.com/v1/posts/${slug}`,
      {
        headers: {
          Authorization: `Bearer ${process.env.MARBLE_API_KEY}`
        }
      }
    );

    if (!response.ok) {
      const error = await response.json();
      
      switch (response.status) {
        case 401:
          throw new Error('Authentication failed: ' + error.message);
        case 404:
          return null; // Post not found
        case 429:
          // Implement retry with backoff
          throw new Error('Rate limit exceeded');
        case 500:
          throw new Error('Server error: ' + error.message);
        default:
          throw new Error(error.message || 'Unknown error');
      }
    }

    return await response.json();
  } catch (error) {
    console.error('Failed to fetch post:', error);
    throw error;
  }
}
```

```python Python
import os
import requests

def fetch_post(slug):
    try:
        response = requests.get(
            f'https://api.marblecms.com/v1/posts/{slug}',
            headers={'Authorization': f'Bearer {os.environ["MARBLE_API_KEY"]}'}
        )
        
        if response.status_code == 401:
            raise Exception(f'Authentication failed: {response.json()["message"]}')
        elif response.status_code == 404:
            return None  # Post not found
        elif response.status_code == 429:
            raise Exception('Rate limit exceeded')
        elif response.status_code == 500:
            error = response.json()
            raise Exception(f'Server error: {error.get("message", "Unknown")}')
        
        response.raise_for_status()
        return response.json()
        
    except requests.exceptions.RequestException as e:
        print(f'Failed to fetch post: {e}')
        raise
```

</CodeGroup>

## Validation errors

When creating or updating resources, validation errors provide detailed feedback:

```json
{
  "error": "Invalid request body",
  "message": "Validation failed",
  "details": [
    {
      "field": "title",
      "message": "Required"
    },
    {
      "field": "slug",
      "message": "String must contain at least 1 character(s)"
    }
  ]
}
```

Each item in the `details` array includes:

- `field` - The field name that failed validation
- `message` - Description of the validation error

## Debugging tips

1. **Check status code first** - The HTTP status code indicates the error category
2. **Read the error message** - The `message` field provides context
3. **Review validation details** - Check the `details` array for field-specific errors
4. **Verify API key** - Ensure you're using the correct key type (public vs private)
5. **Check rate limit headers** - Monitor `X-RateLimit-Remaining` to avoid 429 errors
6. **Enable request logging** - Log requests and responses for troubleshooting

<Warning>
  Never log or expose API keys in error messages, client-side code, or public repositories.
</Warning>
